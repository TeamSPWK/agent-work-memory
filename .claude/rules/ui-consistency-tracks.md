# UI 일관성 + 두 트랙 분리 룰

`web/src/**` (화면·컴포넌트·라우트·CSS) 변경 시 반드시 따른다.
원문은 `docs/DESIGN_SYSTEM.md` §1.0 / §3.4 / §12. 본 룰은 *판단 트리* + *체크리스트*.

## 1. 회피할 4가지 위험

| # | 위험 | 어떻게 터지나 |
|---|------|------------|
| 1 | **트랙 혼동** | Inside-app 화면을 외부 페이지 패턴(중앙 정렬·큰 hero·여백)으로, 외부 페이지를 인사이드앱 패턴(풀폭·데이터 밀도)으로 만듦 |
| 2 | **시안 맹신** | 시안이 *디자인 파트너 데모* + *외부 마케팅 페이지*를 분리 없이 그렸을 때 그대로 이식 — H 라벨·mock 숫자·내부 버전 라벨이 외부에 노출 |
| 3 | **a11y role 불일치** | 같은 패턴(예: tier-grid)에 페이지마다 다른 role(`list` vs `group`) 부여 |
| 4 | **CSS 토큰 분기** | 같은 시각 요소에 페이지별로 inline style + 별개 클래스 작성 — 일관성 깨짐 |

## 2. 작업 시작 전 결정 트리

```
이 화면은 어느 트랙인가?
├─ 인사이드앱 (사용자가 가입 후 보는 운영 화면)
│   → /web/src/screens/** 또는 /web/src/layout/AppShell·OnboardingLayout
│   → 풀폭 + .main padding 24·32 + .page-h eyebrow + .card 카드 패턴
│   → 데이터 밀도 우선, 한 화면 KPI 4 + 테이블 + 비교표 동시 OK
│   → 톤: Calm operations (DESIGN_SYSTEM §2)
│
└─ 외부 페이지 (가입 전 잠재 고객이 보는 마케팅·법무·상태)
    → /web/src/routes/public/** 또는 /web/src/layout/PublicShell
    → .pub-shell + .pub-inner max-width 1120 + .sec padding 96
    → 1컬럼 hero + 큰 typo(h1 64 / h2 36) + 충분한 vertical spacing
    → 톤: 토스 마케팅 패턴 (§12)
    → audience: 모든 사람 (AI 경험자·전문가·초심자·학생), 30초 이해
```

## 3. 외부 페이지 작업 시 체크리스트

새 외부 페이지(또는 외부 페이지 카피·시각 변경) 작업 시 *모든 항목 통과* 필수.

### 3.1 카피 외부 노출 금지 7항 (§12.3)

| 항목 | 금지 예시 | 처리 |
|------|----------|------|
| 내부 가설 라벨 H1·H2·H3·H4 | `H1 회상`·`H4 · 5분 온보딩` tag | 평이한 한국어 (회상·검토·복원·연결) |
| mock vmetric 숫자 (출처 없음) | `62% → 14%`·`62분 → 10분` | 제거 또는 *상황 묘사*로 치환 |
| 내부 버전 라벨 | `v0.1 화면 보기`·`v0.2 · 외부` | 제거 |
| PRD 섹션 번호 | `PRD §1.4 인용`·`(§11.5)` | 제거 |
| 자기 워크스페이스 데이터 가정 | `워크스페이스 자동 점검 5/7 충족` | 가입 전 페이지에서 제거. 후속 sprint에서 *기능 미리보기*로 변환 |
| dev 메타 (PageBand) | 가설 statement + 측정 지표 띠 | `/dev/status`로만 노출 |
| 운영 인원 규모 | `1인 창업자가 만들고 있습니다`·`1인 운영 sustainability` | 운영 인원·키맨 위험은 사용자가 협업 과정에서 알려준 *내부 사실*이지 외부 오픈 요소가 아님. 외부 카피에서는 "운영 정책·응답 시간·SLA"처럼 *결과*만 표현 |

### 3.2 시각·구조 체크리스트

- [ ] 페이지 root 또는 `PublicShell` outlet 안에서 모든 `.sec` 자식이 `.pub-inner`로 wrap되어 있다
- [ ] h1 단일, h2는 섹션마다 1개 (level 2)
- [ ] hero 좌-우 grid 분리 사용 X — 부가 콘텐츠는 hero 아래 별도 `.sec.tight`
- [ ] CTA Link `to=`가 의도와 일치 (예: "어떻게 작동하는지 보기" → flow 섹션이 아니라 가격 페이지로 빠지면 라벨 수정)
- [ ] 같은 패턴(tier-grid, val-grid 등)이 다른 페이지에 있을 때 *a11y role 동일* (Landing의 tier-grid가 `group`이면 Pricing도 `group`)
- [ ] 한국어 평이체, 전문 용어·내부 가설명·mock 수치 없음

### 3.3 audience 카피 검증 (§12.4)

> *"AWM은 AI에게 시킨 일을 1분 만에 다시 설명할 수 있게 해줍니다."* (PRD §4.2)

이 한 줄을 기본으로 각 페이지 카피가 일관되게 변형되어 있는지 확인. AI를 처음
써본 사람·학생도 hero 한 화면에서 30초 안에 *무엇*인지 이해 가능해야 한다.

## 4. 인사이드앱 작업 시 체크리스트

- [ ] `.page-h` eyebrow + h1 + sub 헤더 패턴 유지
- [ ] 좌-우 사이드바 248 + 풀폭 main 유지 — max-width 추가 금지
- [ ] 카드·테이블·KPI grid 자유 — 데이터 밀도 우선
- [ ] role=tablist + role=tab + aria-selected 패턴(Audit·Workspace·Settings 선례) 일관
- [ ] 동일 a11y role을 다른 화면에서 다르게 부여하지 않음

## 5. 두 트랙 충돌 시 결정

같은 컴포넌트(예: `.tier`)가 양 트랙에서 모두 쓰일 때:
- 외부 페이지(`Landing`·`Pricing`)에서는 풀 디스플레이 (큰 typo·여백)
- 인사이드앱(`Billing`·`Members` 등)에서는 그대로 사용하되 카드 패턴(`.card`)
  안에서 자연 정렬되는지만 확인. 외부용 typo 크기를 인사이드앱에 끌고 오지 않음.

## 6. 일관성 깨진 곳 발견 시

- 일관성 깨진 곳 발견 → *즉시 정정* + `docs/DESIGN_SYSTEM.md` §12.3 또는
  관련 섹션에 *왜 깨졌었는지·왜 정정했는지* 1줄 추가 (작은 메타 학습 누적)
- 시안과 production이 어긋날 때 — 시안을 신뢰하기 전에 *시안의 의도*(누구를
  위한 화면인가)를 사용자에게 한 번 확인. 시안이 데모용이면 production 외부
  페이지로 그대로 이식하지 않음.

## 7. 적용 대상

- `web/src/**` 모든 변경 (TSX·CSS·시드)
- `docs/DESIGN_SYSTEM.md` 직접 수정 시 본 룰도 동기 갱신
- 시안 → 코드 이식 작업 (`docs/projects/plans/p0-design-v0*/` → `web/src`)
- 외부 페이지 카피·콘텐츠 변경
- 새 컴포넌트 추가 시 — 기존 패턴 재사용 우선, 신규 패턴은 §1·§3.4와 호환되는지 확인
