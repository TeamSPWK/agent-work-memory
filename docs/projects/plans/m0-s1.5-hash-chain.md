# M0/S1.5 — Hash Chain CLI 최소 구현

> Sub-plan of `docs/projects/plans/m0-tech-hypothesis-validation.md` (M0 Tech Hypothesis Validation).
> Scope: PRD §5.5 SHA-256 chain + verify CLI. Export(PDF/CSV/JSON)는 분리 sprint.

## Context

- PRD §5.5(Audit Layer v2.0 MVP)는 **SHA-256 해시 체인 + 무결성 검증 명령**을 요구한다. *"각 이벤트가 직전 이벤트 해시를 포함, 무결성 검증 명령 제공"*.
- v1 자산 회수(`tech-validation/m0` branch)에서 `bin/awm.mjs recordEvent`(L228-254)는 prevHash/hash 없이 events.jsonl append만 한다. `awm help`에 verify/audit/integrity 명령 없음.
- M0/S1.6 baseline 7 발견 중 **#4 Audit hash fake** — `web/src/lib/useIngest.ts toAuditEvent`가 `hash: id.slice(-8)`, `prev: '--------'`, `type: 'session.end'`를 더미로 채워 Audit Trail UI에서 *변조 불가성*이 거짓 표현됨.
- M0/S3 H2-b 측정 기준 *"hash chain 변조 5/5 탐지"*의 전제 — S1.5 구현이 없으면 측정 불가.

## Problem (MECE)

1. **데이터 모델 결손** — events.jsonl 한 줄당 prevHash/hash 컬럼 부재.
2. **검증 도구 결손** — `awm audit verify` 명령 부재.
3. **기존 데이터** — 본인 캡처 30 세션이 hash 없는 events.jsonl로 누적된 상태. 측정 baseline을 보존하려면 *rebuild*가 필요.
4. **UI 노출** — Audit Trail이 fake hash를 표시 중. 실제 chain 결과(hash·prev·broken)로 교체 필요.
5. **회귀 위험** — recordEvent가 자주 호출되는 hot path(매 hook 이벤트). hash 계산 추가가 latency·실패 모드를 늘리지 않게 해야 함.

## Solution

### S1. `bin/hashchain.mjs` 신규 (~80줄)

순수 helper. 외부 의존 없음.

```
GENESIS_PREV = "0".repeat(64)
canonicalize(event) — JSON.stringify with deterministic key order, exclude {hash, prev}
computeHash(prev, event) → SHA-256(prev + canonicalize(event)) hex (64 char)
verifyChain(events[]) → { ok, brokenAt, brokenReason, total }
  - 첫 이벤트 prev === GENESIS_PREV
  - 각 이벤트의 hash === computeHash(이전 hash, 이벤트)
  - prev field가 직전 hash와 일치
  - 실패 시 첫 깨진 인덱스 + 사유 반환
```

### S2. `bin/awm.mjs recordEvent` 수정 (~15줄 추가)

- append 직전, events.jsonl tail에서 마지막 hash 1줄 읽어 `prev` 결정 (없으면 GENESIS_PREV).
- `event.prev = prev`, `event.hash = computeHash(prev, event)` 추가.
- 기존 atomic-write 패턴 유지(append-only, fsync는 v2.x).
- tail 읽기 실패 시: GENESIS_PREV로 폴백하지 않고 *명시적 에러*. (chain 깨짐을 silent로 시작하면 검증 의미 없음)

### S3. `bin/awm.mjs` dispatcher + printHelp (~30줄)

3 명령 추가:
- `awm audit verify` — 전체 events.jsonl 읽어 verifyChain 호출, 결과 표시 (exit 0 / 1)
- `awm audit rebuild` — 기존 events 순서대로 prev/hash 재생성하여 events.jsonl 덮어쓰기. 백업 `events.jsonl.pre-chain-<ISO>` 보존
- `awm audit show --last N` — 최근 N건의 id/prev/hash/createdAt 표시 (디버깅)

printHelp에 위 3 명령 + S1.5 note 1줄.

### S4. `tests/hashchain.test.mjs` 신규 — 변조 5 case + 정상 3 case

| # | case | 기대 |
|---|------|------|
| 1 | genesis 단일 이벤트 | ok |
| 2 | 정상 3 이벤트 chain | ok |
| 3 | 정상 100 이벤트 chain | ok |
| 4 | 필드 수정 (1번째 이벤트 summary 변경) | broken at 1 |
| 5 | 레코드 삽입 (중간에 fake 이벤트 끼움) | broken at 삽입 위치 |
| 6 | 삭제 (1번째 이벤트 제거) | broken at 0 (prev 불일치) |
| 7 | 순서 바꾸기 (2-3 swap) | broken at 2 |
| 8 | hash 직접 조작 (hash 필드만 변경) | broken at 해당 위치 |

변조 5/5(case 4~8) — S3 H2-b 측정 기준 *"변조 5/5 탐지"*를 unit test로 사전 충족.

### S5. `web/src/lib/useIngest.ts toAuditEvent` 수정 (~5줄)

기존:
```ts
hash: id.slice(-8),
prev: '--------',
broken: false,
type: 'session.end',
```

수정:
```ts
hash: pkt.hash ?? '(미해시)',
prev: pkt.prev ?? '--------',
broken: pkt.broken ?? false,  // verify CLI 결과 반영
type: pkt.eventType ?? 'session.end',
```

`bin/awm.mjs ingestSessions`에서 events.jsonl의 hash/prev/broken을 ingest 응답에 포함해야 함(S6 작업).

### S6. `bin/awm.mjs ingestSessions` 보강 (~5줄)

workPackets 생성 시 마지막 이벤트의 hash·prev·broken을 함께 출력. broken은 verify 결과를 reuse(또는 false default).
