## Problem

5 화면을 한 sprint에 묶으면 diff 1500+줄 → review·Evaluator 검증 부담 폭증. S2.5(.a/.b/.c) 패턴이 H1에서 PASS 받은 선례 → 동일 패턴 적용.

| # | 요구 | 측정 |
|---|---|---|
| 1 | 시각 정합성 | `h3-incident.jsx` 5 함수의 KPI·tile·canvas·split·timeline·match-line 시각 1:1 재현 |
| 2 | 라우팅 | `/risk` 단일 + `/incidents/:id` + `?tab=replay\|event\|reviewer\|note` (기본 `replay`) |
| 3 | 시드 분리 | risk(RISK_CATEGORIES 8 + RISK_SIGNALS 7) ↔ incident(INCIDENT 단일) 두 파일 |
| 4 | cross-link 흐름 | Risk Radar 사고 cta → /incidents/:id ; Replay 후보 클릭 → ?tab=event ; Event detail → ?tab=reviewer ; Reviewer Brief → Audit row 등 |
| 5 | Audit/Sessions 데이터 일관성 | INCIDENT.events 의 hash·session·repo가 `seed/audit.ts` + `seed/sessions.ts` + `seed/sessionDetail.ts` 의 id (s-024, s-021, ev-2401)와 일치 |
| 6 | 사고 mock 한계 | system 감지 이벤트(e5/e6/e7)는 *외부 메트릭 mock* — 시각 한계 배지 |
| 7 | 큰 인터랙티브 | incident-canvas 2D timeline (가로축 분 + 세로축 카테고리 row + clickable mark + axis labels). 기존 CSS(`incident-canvas`, `ic-row`, `ic-mark`, `ic-axis`) 재사용 |
| 8 | 테스트 | 각 sub-sprint별 vitest+RTL 케이스 4~6건 — 누적 19 → 30~35 |

**예상 8 파일** (복잡도 8 — 복잡 경계).

**핵심 결정**:
- 4탭 wrapper(Incident.tsx)는 *2개만 구현하고 나머지 2개는 PlaceholderScreen 인라인*. S2.7.c에서 교체.
- Replay의 `incident-canvas` 2D timeline은 `xPct(x)` 변환 + sevSize map 그대로 이식. selected state로 우측 detail 갱신.
- Note의 timestamp 자동 생성 (`new Date()`)은 디자인 라이브 동작 그대로 — 테스트에서 `vi.useFakeTimers()` 또는 mock skip.

**Exit**:
1. 라우팅: `/incidents/INC-26-014` = Replay 기본, `?tab=note` 노트
2. 잘못된 tab 값 → replay fallback
3. canvas mark 클릭 시 우측 detail 갱신
4. 3 bucket (likely/verified/unknown) 표시
5. Note 메모 추가 시 timeline 갱신

7. 사고 mock 한계 배지(system 감지 이벤트가 외부 메트릭 mock이라는 안내)
8. typecheck + vitest 통과

### 공통 결정

#### 라우팅 패턴
- `/risk`: 단일 라우트
- `/incidents/:id`: 단일 라우트 + `?tab=replay|event|reviewer|note` (Audit과 동일 useSearchParams 패턴)
- 기본 tab = `replay`. 잘못된 값 → `replay` fallback

#### Incident id 처리
- 디자인 lock은 `INC-26-014` 단일. URL `/incidents/INC-26-014`로 고정.
- 다른 id는 SessionDetail s-024 외 fallback 패턴 답습 — mock 한계 배지 + 단일 INCIDENT 데이터 표시.

#### 시드 분리
- `lib/seed/risk.ts`: RISK_CATEGORIES + RISK_SIGNALS 만
- `lib/seed/incident.ts`: INCIDENT 만 (events·rows·buckets·notes 포함)
- 두 시드는 **분리 유지** — Risk Radar는 incident을 import 안 함 (사고 cta는 string id만 사용)

#### 사고 mock 한계
- INCIDENT.events 의 e5/e6/e7 (system: true) = "외부 메트릭 감지" 위장 — Datadog/엣지 모니터 등 *실제 미연동*
- IncidentReplay 또는 Note 한곳에 *"system 감지 이벤트는 외부 메트릭 통합 후 실측"* 배지

#### Audit/Session cross-id 정합
- INCIDENT.events 의 `session` 값(s-021/s-022/s-024)이 `seed/sessions.ts` 와 일치 ✓
- AuditTrail의 e3 evId `ev-2401` ↔ `seed/audit.ts` aud-001 (sha 3acb09d)이 같은 이벤트인지 확인 — 시드 시점 PR에서 *id 재사용 권고* 검토

#### 안 함
- 실 incident 생성/CRUD (m2 S5 Auth 이후)
- system 메트릭 실 연동 (별도 sprint)
- Postmortem 실 PDF (S6)
- INC-26-014 외 incident (single mock)
