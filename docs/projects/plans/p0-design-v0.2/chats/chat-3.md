# Decision record — v0.2 외부 페이지 (chat-3)

날짜 2026-05-11 · 의사결정자 jay (Spacewalk · 1인 창업자) · 시안 버전 v0.2

---

## 배경

v0.1까지는 가입 이후의 가설 3종 (H1 회상 · H2 감사 · H3 1차 원인) + H4 온보딩 + Workspace + Settings, 총 5 그룹 22 화면이 완성됐다.
v0.2의 작업 단위는 **미가입자가 도달하는 외부 페이지 13종**이다. 가입 이전에 사용자가 만나는 모든 면이 대상이다.

> 외부 13종 = 랜딩 · 가격 · 회원가입 · 로그인 · 비밀번호 재설정 · 이용약관 · 개인정보처리방침 · 환불 정책 · 사업자 정보 · 회사 · 404 · 500 · 점검 · 상태 페이지

## 의사결정 1 — 사이드바와 페르소나 토글을 외부 페이지에서 끈다

**문제** v0.1 shell은 좌측 사이드바 (그룹 + 화면) + 상단 페르소나 토글 + 워크스페이스 셀렉터를 전제로 한다. 미가입자에게는 이 세 가지가 모두 의미 없다.

**결정** Public 그룹만 별도의 *flat top-bar + footer* shell (`src/public-shell.jsx`)을 쓴다. 사이드바·페르소나·워크스페이스 셀렉터는 렌더하지 않는다. 가설 배너도 hypothesis 단위가 아니라 *page 단위* (`PUBLIC_HYPS`)로 바꾼다.

**근거** PRD §1·§5의 외부 페이지 정의는 명시적으로 "사이드바 없음 · 페르소나 없음 · 검증 단위는 랜딩 / 가격 / 가입 3 페이지". 시안에서 사이드바를 남기면 가입 후 화면과 외부 페이지를 시각적으로 같이 본 사람이 "사이드바가 미가입자에게도 뜬다"고 오해할 수 있다.

**구체화** Public Pages 그룹의 사이드바 옵션은 시안 *navigation 용*으로만 남긴다. 실제 사용자에게는 사이드바 자체가 없다. Shell은 sidebar group의 `external: true` 플래그로 분기한다.

## 의사결정 2 — 가설 배너는 page 단위 (landing / pricing / signup만)

**문제** v0.1은 hypothesis 단위 가설 배너를 모든 화면 위에 띄웠다. 외부 13개 중 가설 검증 대상은 *랜딩 · 가격 · 가입* 3개 뿐이다.

**결정** `PUBLIC_HYPS = { landing, pricing, signup }`만 가설 배너 (검증 지표 + from→to)를 띄운다. 나머지 10개는 *"Public · 외부 페이지 · 가설 검증 대상 아님"* 회색 배너로 통일.

**근거** 검증 가설이 없는 페이지에 가설을 가짜로 채우면 시안이 검증 대시보드로서의 신뢰를 잃는다. 법무 4종·회사·에러 3종·상태 페이지는 가설 검증 대상이 아니라 *가입 funnel의 신뢰 인프라*다.

## 의사결정 3 — 외부 페이지에 가상 인물·회사명 금지 룰을 유지

**문제** 랜딩의 "왜 지금인가" 섹션은 v0.1 가설 데이터 (`s-021` / 인디 개발자 박재현 등) 와 분리해야 한다. 미가입자가 보는 페이지에서 가상 인물·회사명을 인용하면 신뢰가 무너진다.

**결정** `PUBLIC_NEWS` 4건은 **출처 chip이 붙은 외부 보도·연구 사례만** 인용한다.

- Replit Agent · 1,206명 데이터 (Newsweek 2025-07)
- PocketOS · Cursor 9초 삭제 (인디 개발자 X thread 2025-09)
- UK 학생 33% AI 작성 (BBC 2025-02)
- METR · 체감 vs 실측 19% 느림 (METR study 2025-03)

**근거** PRD §1.4에 동일 4건이 이미 인용돼 있다. 외부 페이지는 *PRD의 외부 출처만* 노출하고, 내부 가설 케이스 (s-021 / s-024 / INC-26-014)는 가입 후 화면에서만 등장한다.

**구체화** 사고 히스토리 (상태 페이지)도 *외부 의존성 사고* (OpenAI · PopBill · GitHub Webhook)만 다룬다. v0.1 사건 식별자와는 cross-link 안 한다.

## 의사결정 4 — 가격 페이지의 결제 단위는 Active Operator로 고정

**문제** SaaS 가격을 *사용자 수* (seat)로 부과하면 Reviewer를 적게 두라는 인센티브가 생긴다. Reviewer가 줄면 H2 감사 가설이 무너진다.

**결정** 결제 단위는 **Active Operator** — 지난 30일 1회 이상 AI 작업이 기록된 사용자. Reviewer·Admin은 활동량과 무관하게 *항상 무료*.

**구체화** 가격 페이지·랜딩 §7 가격 미리보기·이용약관 §2·환불 정책에 **동일 정의 문구를 4지점 반복**한다. PRD §11.2의 "결제가 Reviewer 수를 깎지 않게 한다" 원칙과 일치.

**디자인 파트너 50%** 는 Team 티어에만 chip으로 노출 (₩200,000 → ₩100,000). 기간 한정 아닌 *선착순 5팀*. 격주 인터뷰 1회를 계약 조건으로.

## 의사결정 5 — 법무 4종은 "법무 자문 후 작성" 시안

**문제** v0.2는 시안이지 실제 법무 문서가 아니다. 그러나 항목·헤더가 누락되면 외부 페이지가 *완전한 가입 funnel*로 보이지 않는다.

**결정** 법무 4종 (이용약관·개인정보처리방침·환불·사업자 정보)은 다음 4가지만 노출:

1. **TOC** (제1조·제2조 헤더 목록)
2. **핵심 원칙 callout** — 1인 운영 sustainability·원문 transcript 미저장·§27 권고 보존 항목 같은 *AWM 고유 정책*은 본문에 명시
3. **표 형식 핵심 데이터** — Active Operator 정의·보존 기간·환불 3-column 등 *기계적 정보*는 표로
4. **본문 placeholder** — "법무 자문 후 작성 예정 (docs/areas/regulatory/legal-pages.md)" 회색 박스

**근거** 법무 문구는 변호사 자문이 끝나기 전에 시안에 박아두면 *나중에 그대로 배포될 위험*이 있다. 시안에서 법무 문구가 placeholder인 것을 명확히 보이게 한다.

## 의사결정 6 — 회사 페이지는 4 블록 솔직 톤

**문제** "1인 운영"을 마케팅 카피로 포장하면 첫 사고에서 신뢰가 무너진다 (METR study가 시사한 *체감 vs 실측 괴리* 와 같은 패턴).

**결정** 회사 페이지는 4 블록 구조:

1. **왜 1인인가** — "2026년에 가능해진 시점"이라는 사실 그대로
2. **어떻게 보완하는가** — self-serve / 채널톡 / 자동 응답 / 상태 페이지 / 무음 시간대
3. **무엇을 안 하나** — 24/7 SLA / 엔터프라이즈 영업 / 전화·실시간 채팅 / 다국어 / PR 키트 / 컨설팅
4. **무엇이 무너질 수 있나** — 키맨 위험 · 번아웃 · 이탈 → data export / source escrow / 월 운영 보고

**근거** PRD §11.3과 동일 카피. 디자인 파트너 계약서에도 같은 문장이 들어간다.

## 의사결정 7 — 상태 페이지는 무음 시간대도 즉시 갱신

**문제** 1인 운영은 *24/7 사람의 응답*을 보장하지 못한다. 그렇다고 *24/7 가시성*까지 포기하면 신뢰가 무너진다.

**결정** 상태 페이지는

- 5분 폴링 자동 모니터링 (자체 + 외부 의존성)
- 무음 시간대 (밤 9시 ~ 오전 8시) 사고도 즉시 갱신
- 매월 1일 uptime · 응답 시간 · 결제 잔액 공개
- 사고 히스토리는 *외부 서비스* 위주

**구체화** 시안의 사고 히스토리 3건은 모두 외부 의존성 (PopBill · OpenAI · GitHub Webhook). v0.1 내부 사건 (INC-26-014 등)과 cross-link 안 함.

## 의사결정 8 — 가입 후 자동 점프는 H4 화면 1

**문제** 가입 페이지에서 가입 직후 어디로 갈지 명시되지 않으면, 가입 완료가 *first-value 도달*과 분리된다.

**결정** 가입 페이지의 primary CTA 문구를 **"가입 후 H4 화면 1로"** 로 고정. 5분 first-value 미리보기 패널을 가입 우측에 항상 노출. 가입 = "5분 first-value의 0번째 단계" 로 정의.

**근거** PRD §3 first-value 정의 — 가입에서 Today 화면까지 ≤ 90초 (median). 가입 페이지 자체가 *진행 게이지 0 / 5*. 다음 화면이 *1 / 5*.

## 의사결정 9 — 사업자 정보 미입력 슬롯은 회색 placeholder

**문제** 사업자 등록·통신판매업 신고는 디자인 파트너 모집 *후* 진행될 가능성이 높다. 시안 단계에서 가짜 번호를 박아두면 안 된다.

**결정** 사업자등록번호·통신판매업 신고번호·사업장 주소는 **`[사업자 등록 후 입력]`** 회색 텍스트로 비워둔다 (`PUBLIC_BIZ.bizNo = ""`).

**구체화** Footer·사업자 정보 페이지 두 군데에서 같은 placeholder를 쓴다. 실제 번호가 들어가면 한 곳만 수정해도 두 곳이 동기화된다 (`PUBLIC_BIZ` single source).

## 의사결정 10 — chat-1 / chat-2 / chat-3 로 의사결정 기록 분리

**문제** v0.1까지의 의사결정 12 + v0.2 의사결정 10 = 22 개를 한 파일에 두면 결정 단위 *추적 가능성*이 무너진다.

**결정** `chats/chat-1.md` (v0.0 가설 3종 시안), `chats/chat-2.md` (v0.1 가입 후 17 화면), `chats/chat-3.md` (이 파일 · v0.2 외부 13 화면) 로 분리. 각 파일은 *그 chat 세션의 의사결정*만 담는다.

**다음 chat** chat-4는 D2 (디자인 파트너 모집 단계)에서 진행될 예정. 주요 안건은 (a) 법무 자문 결과 반영, (b) 결제 토스페이먼츠 실연동 시안, (c) 다국어 (en) 검토.
